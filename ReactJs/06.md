# React — Form Handling

Handling forms in React means capturing what the user types and reflecting it in state. This module walks through **three progressively better approaches** — from brute-force logging to a single, reusable `handleChange` function.

---

## 1. Approach 1 — Brute Force (Just Reading Values)

```jsx
<input onChange={(e) => { console.log(e.target.value) }} />
<input onChange={(e) => { console.log(e.target.value) }} />
<input onChange={(e) => { console.log(e.target.value) }} />
```

- Every input has its own `onChange` handler that just logs the typed value.
- This is called **single-way binding** (or "uncontrolled-ish" reading) — the input manages its own value internally in the DOM; React is only *listening* to changes, not driving the input's displayed value.
- **Problem:** React doesn't actually **store** or **know** these values anywhere — they exist only in the console log, not in component state. There's no way to use this data elsewhere (validation, submission, conditional rendering, etc.).

---

## 2. Approach 2 — One `useState` Per Field

```jsx
const [name, setName] = useState("");
const [email, setEmail] = useState("");
const [password, setPassword] = useState("");

<input onChange={(e) => setName(e.target.value)} />
<input onChange={(e) => setEmail(e.target.value)} />
<input onChange={(e) => setPassword(e.target.value)} />
```

- Now **React actually knows** the value of each field, because it's stored in state.
- This creates **controlled inputs** — the input's displayed value is driven by React state (in a fuller implementation you'd also set `value={name}` on the input), and every keystroke flows back into state via `onChange`.
- **Problem — doesn't scale:**
  > **As many inputs, that many states.** A 3-field form needs 3 `useState` calls, 3 setter functions, and 3 separate `onChange` handlers. A 10-field form would need 10 of each — repetitive and hard to maintain.

---

## 3. Approach 3 — A Single Object State (`formData`)

Instead of one state variable per field, store **all fields in one object**:

```jsx
const [formData, setFormData] = useState({
  name: "",
  email: "",
  password: ""
});

<input onChange={(e) => setFormData({ ...formData, name: e.target.value })} />
```

### Why the spread (`...formData`) is essential

```js
setFormData({ ...formData, name: e.target.value })
```

- `{ ...formData, name: e.target.value }` creates a **brand-new object** — a copy of `formData` placed at a **new memory location (new reference)** — with every existing key/value carried over, except `name`, which is overwritten with the new typed value.
- Because it's a new object reference, React detects the state actually changed and re-renders correctly (see the earlier notes on primitive vs reference state — this is exactly that rule in action).
- If you forgot the spread and wrote `setFormData({ name: e.target.value })`, you'd **wipe out** `email` and `password` entirely, since the new object would only contain `name`.

**Still a problem:** you'd need to hand-write a near-identical `onChange` for every field (`name: e.target.value`, `email: e.target.value`, `password: e.target.value`) — still repetitive, just less repetitive than Approach 2.

---

## 4. Approach 4 — Optimized: One Reusable `handleChange`

Instead of writing a unique inline `onChange` per input, extract **one function** that works for *any* field, using the input's own `name` attribute to know which key to update.

```jsx
const [formData, setFormData] = useState({
  name: "",
  email: "",
  password: ""
});

const handleChange = (e) => {
  setFormData({ ...formData, [e.target.name]: e.target.value });
};

<input name="name" onChange={handleChange} />
<input name="email" onChange={handleChange} />
<input name="password" onChange={handleChange} />
```

### The Key Trick — Computed Property Names

```js
{ ...formData, [e.target.name]: e.target.value }
```

- `[e.target.name]` is a **computed property name** — JavaScript evaluates whatever is inside the brackets and uses *that result* as the actual key name in the new object.
- Since each `<input>` has its own `name` attribute (`"name"`, `"email"`, or `"password"`), the **same** `handleChange` function correctly updates the **correct** key every time, just by reading `e.target.name` at runtime.
- This is what makes the function **generic and reusable** across every field in the form — no matter how many inputs you add, you never need a new handler.

### Cleaner Version — Destructuring `e.target`

```js
const handleChange = (e) => {
  const { name, value } = e.target;
  setFormData({ ...formData, [name]: value });
};
```

- `const { name, value } = e.target` pulls the two properties you actually need directly out of the event's target element, making the function body a little more readable than repeating `e.target.x` twice.

---

## 5. Comparing All Approaches

| Approach | State Shape | Scales Well? | Reusable Handler? |
|---|---|---|---|
| 1. Brute force | None (just `console.log`) | No — no state stored | No |
| 2. One state per field | `name`, `email`, `password` as separate `useState` calls | No — N fields = N states | No — separate handler per field |
| 3. Single object, inline handlers | One `formData` object | Better — one state for the whole form | No — still one inline handler per field |
| 4. Single object + `handleChange` | One `formData` object | ✅ Yes | ✅ Yes — one function for all fields, driven by `name` attribute |

## 6. Code Walkthrough

```jsx
import React, { useState } from "react";

const App = () => {
  const [formData, setFormData] = useState({});

  const handleChange = (e) => {
    const { name, value } = e.target;
    const updated = { ...formData, [name]: value };
    setFormData(updated);
    console.log(updated);
  };

  return (
    <div className="form">
      <input
        name="name"
        type="text"
        placeholder="Enter Your Name"
        onChange={handleChange}
      />
      <input
        name="email"
        type="text"
        placeholder="Enter Your Email"
        onChange={handleChange}
      />
      <input
        name="password"
        type="text"
        placeholder="Enter Your Password"
        onChange={handleChange}
      />
      <button className="btn">Submit</button>
    </div>
  );
};

export default App;
```

### Breakdown

- **`formData` starts as `{}`** (empty object) rather than pre-defining `name`/`email`/`password` as empty strings. This still works because computed property names simply **add** new keys to the object the first time each input fires `onChange` — but note that until a field is touched, its key won't exist in `formData` at all (so reading `formData.email` before typing anything returns `undefined`, not `""`).
- **On every keystroke** in *any* input:
  1. `handleChange` fires with the native event `e`.
  2. `{ name, value } = e.target` extracts which field fired (`e.target.name`, e.g. `"email"`) and what was typed (`e.target.value`).
  3. `updated = { ...formData, [name]: value }` builds a new object — all previous fields preserved, plus the current field updated (or added, if it's the first keystroke in that field).
  4. `setFormData(updated)` — new reference → React re-renders.
  5. `console.log(updated)` prints the full, up-to-date form object after every keystroke — handy for debugging what's being captured.
- **The `Submit` button** currently has no `onClick`/`onSubmit` logic attached — in a complete implementation, you'd typically wrap the inputs in a `<form onSubmit={handleSubmit}>` and call `e.preventDefault()` inside `handleSubmit` to stop the default full-page reload, then do something with the final `formData` (e.g. send it to an API).

### Styling Notes (`index.css`)

- `.form` is a centered flex column with a white border and rounded corners, holding the inputs and button with consistent `20px` gaps.
- Inputs are `300px` wide, `40px` tall, with a transparent border by default; on `:focus` the border turns green (`#50C878`) and gains a soft glow via `box-shadow`, giving clear visual feedback of the active field.
- `.btn` is a full-width green submit button with the same `scale(0.95)` press animation on `:active` seen in earlier examples, for consistent tactile feedback across the app.

## Quick Summary

```
Brute force        →  onChange logs value only, nothing stored
One state/field    →  works, but N fields = N states (doesn't scale)
Object + spread     →  one state, {...formData, key: value} → new reference → rerenders
Object + handleChange →  ONE reusable function using [e.target.name] as a computed key

Core trick: [e.target.name] lets a single handleChange work for every input,
because JS evaluates the bracketed expression into the actual object key at runtime.

Always spread the previous state ({ ...formData, ... }) when updating
one field of an object in state — otherwise you'll silently lose the other fields.
```
