## 3. Export Default vs Named Exports

| | `export default` | Named export (`export const X`) |
|---|---|---|
| Import name | Can be imported under **any name you choose** | Must be imported using the **exact same name** it was exported with (unless aliased with `as`) |
| Count per file | Only **one** default export per file | **Multiple** named exports allowed per file |
| Import syntax | `import Anything from './App'` | `import { App } from './App'` |

```js
// default export
export default App;
import MyApp from './App';        // ✅ any name works

// named export
export const helperFn = () => {};
import { helperFn } from './utils';   // ❌ must match the exported name
import { helperFn as h } from './utils'; // ✅ unless aliased
```

## 4. Hooks — What Makes React "React"

**Hooks** are special functions that let functional components "hook into" React features (state, lifecycle, context, etc.) — they're the reason functional components can do everything class components used to do, without needing `this` or class syntax.

> Hooks are the core mechanism behind React's modern, functional-component-first design.

---

## 5. `useState` Hook

`useState` lets a component:
1. **Hold/store** a piece of state (a variable that persists across re-renders).
2. **Trigger a re-render** whenever that state is updated.

### Basic call

```js
let result = useState();
console.log(result); // [undefined, function]
```

`useState()` always returns an **array with exactly 2 items**:
1. The current **state value**
2. A **setter function** to update it

Using **array destructuring**, this is conventionally written as:

```js
const [state, setState] = useState(0);
```

| Part | Meaning |
|---|---|
| `useState(0)` | `0` is the **initial value** passed to the hook |
| `state` | The **current value** of the state |
| `setState` | A **function** used to update `state` and trigger a re-render |

---

## 6. What `setState` Actually Does

The setter function returned by `useState` has **two jobs**:

1. **Updates the state value** (schedules the new value to replace the old one).
2. **Triggers a re-render of the component** that owns this state — internally, this works via **closures** and **higher-order functions (HOFs)**: the setter function "remembers" (closes over) which component instance's state it belongs to, and calling it tells React to re-run that component function with the updated value.

### Hard Rules

- **`setState` (and all hooks) can only be called inside a functional component** (or inside a custom hook, which itself is only usable inside a component). You cannot call it globally, in a regular utility function, or conditionally at the top level — this is a fixed rule of React ("Rules of Hooks"), not a stylistic suggestion.
- Hooks must be called in the **same order on every render** — never inside loops, conditions, or nested functions.

---

## 7. Important `setState` Behavior — Bail-Out on Same Value

> If you call `setState` with the **exact same value** the state currently holds, React will **not** trigger a re-render.

This is called **bailing out** of a render. React uses `Object.is()` comparison internally to check the new value against the current one:

- If `Object.is(newValue, oldValue)` is `true` → React skips re-rendering that component (it still updates the underlying state reference, but no render/commit happens, and your component function doesn't run again).
- If the values differ → React proceeds with the render phase and commit phase.

**Why this matters:** it's a built-in performance optimization — no point in re-rendering the UI if nothing actually changed.

⚠️ **Important nuance for objects/arrays:** this comparison is by **reference**, not deep equality. So `setState({ ...state })` (a *new* object with the *same* values) **will** trigger a re-render, because it's a different object reference — even though the contents look identical.

## 8. Code Walkthrough

```jsx
import React, { useState } from 'react'

const App = () => {
  const [count, setCount] = useState(0);
  console.log(count);

  const [flag, setFlag] = useState(true);
  console.log(flag);

  return (
    <div className='app'>
      <span>{count}</span>
      <h1>VariableCount</h1>
      <button className='inc' onClick={() => setCount(count + 1)}>Increment</button>
      <h2 className="ftext">{flag.toString().toUpperCase()}</h2>
      <button className='flag' onClick={() => setFlag(false)}>†</button>
    </div>
  )
}

export default App
```

### Breakdown

- **`count` state** — starts at `0`. Clicking **Increment** calls `setCount(count + 1)`, which is always a *new* number, so React re-renders every click and `<span>{count}</span>` updates.
- **`flag` state** — starts at `true`, displayed as `"TRUE"` (via `.toString().toUpperCase()`). Clicking the `†` button calls `setFlag(false)`.
  - **First click**: `true` → `false`. Values differ → **re-renders**, UI updates to `"FALSE"`.
  - **Every click after that**: `setFlag(false)` is called again, but `flag` is *already* `false`. Since `Object.is(false, false)` is `true`, React **bails out** — no re-render happens, no `console.log(flag)` fires again for that update, and the UI stays exactly as `"FALSE"`.
- This demonstrates the same-value bail-out rule directly: `setCount` keeps triggering renders (value always changes), while `setFlag(false)` only renders **once**, then silently does nothing on repeated clicks with the same value.

---

## Quick Summary

```
useState(initial) → [state, setState]

setState(newValue):
   if newValue === current state (Object.is) → SKIP render (bail-out)
   else → update state → re-render component (render phase → commit phase)

Hooks Rule: only callable inside functional components, same order every render
```
