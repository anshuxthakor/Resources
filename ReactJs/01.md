# ReactJS — Origins, Virtual DOM & Reconciliation Notes

## 1. What is ReactJS?

React is a **frontend JavaScript library** (not a full framework) used to build **SPAs (Single Page Applications)** — apps where a single HTML page is dynamically updated instead of the browser reloading a new page for every navigation.

- Created by **Jordan Walke**, a software engineer at **Facebook**.
- Internally, it started as a prototype called **"FaxJS"** (around 2011), which experimented with the idea of a **Virtual DOM** — a lightweight in-memory copy of the real DOM.
- It was **open-sourced in 2013** under the name **React**.
- Core idea from day one: don't touch the slow real DOM directly for every small change — update a lightweight virtual copy first, figure out the minimal changes needed, then apply only those changes to the real DOM.

---

## 2. Real DOM vs Virtual DOM

| Real DOM | Virtual DOM |
|---|---|
| The browser's actual document tree | An in-memory, lightweight JS object copy of the DOM |
| Updating it is **expensive** (triggers layout/reflow & repaint) | Updating it is **cheap** (just JS objects) |
| Directly visible to the user | Not rendered — used internally for comparison |
| Large / heavy | Small / lightweight |

**Flow:** Real DOM element → `ReactDOM` takes it as a container → converted into a tree of nodes via `ReactDOM.createRoot(realDomElement)` → this becomes the **root node** React manages.

```js
const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(<App />);
```

## 3. Reconciliation — How React Updates the UI

**Reconciliation** is React's process for figuring out *what changed* and updating the UI efficiently. It runs in **two phases**:

### Render Phase (steps 1–5)
1. A **new Virtual DOM (NVD)** tree is created (based on new state/props).
2. React compares this NVD with the **previous Virtual DOM**.
3. This comparison is done using the **diffing algorithm** — it walks both trees and figures out the minimal set of differences.
4. The differences (changes) are calculated and **stored** in a list of effects.
5. *(This entire phase can be paused, interrupted, or thrown away — it doesn't touch the screen yet.)*

### Commit Phase (steps 6–7)
6. React applies the stored changes to the **real DOM**.
7. The browser performs **painting** — rendering the updated real DOM visually on screen. This is what the user actually sees.

> **Why split into two phases?** The render phase is *pure calculation* (safe to pause/restart), while the commit phase *actually touches the DOM* (must run synchronously, uninterrupted). This separation is what makes concurrent features possible.

**The cycle repeats:** After the real DOM is updated, any further state/prop change triggers a brand-new Virtual DOM, and the whole render → commit cycle runs again.

```
State/Props change
  —> Render Phase
   → New Virtual DOM created
   → Diffed against previous Virtual DOM   
   → Changes calculated & stored

  —> Commit Phase                                         
   → Real DOM updated                      
   → Browser paints the screen
   
   → (repeat on next update)
```

## 4. Reconciliation Engines: Stack vs Fiber

| | Stack Reconciler | Fiber Reconciler |
|---|---|---|
| Used | React's original engine (2013–2016) | Introduced in **React 16 (2017)**, a full rewrite |
| Nature | Synchronous, recursive — once it started diffing, it couldn't be paused | Asynchronous & **incremental** — work can be split into units, paused, resumed, or abandoned |
| Big update problem | Large updates could block the main thread, causing UI jank | Splits work across frames, allowing high-priority updates (e.g. user typing) to interrupt low-priority ones |
| Enables | Basic virtual DOM diffing | Concurrent features, Suspense, prioritization of updates |

**Flow across React's history:**
```
Real DOM → Virtual DOM → Fiber Tree (the current internal data structure React diffs against)
```

The **Fiber tree** is essentially the modern implementation detail behind the Virtual DOM — each "fiber" is a JS object representing a unit of work for a component, which lets React schedule and prioritize rendering work.

## Quick Summary

```
FaxJS (2011, internal prototype)
   → React open-sourced (2013) by Jordan Walke @ Facebook
   → Stack Reconciler (original diffing engine)
   → Fiber Reconciler (React 16, 2017 — incremental, interruptible rendering)

Every update:
State/Props change → New Virtual DOM → Diff (Render Phase) → Apply + Paint (Commit Phase) → repeat
```
