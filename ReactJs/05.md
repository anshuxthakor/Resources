# React — HMR, Bundlers & Primitive vs Reference State

## 1. HMR — Hot Module Replacement (Recap)

**HMR** continuously monitors your project files and, whenever something changes, rebuilds only the affected part of the **DOM tree** — instead of reloading the whole page.

- Behind HMR (and the dev/build process in general) sit **bundlers** — tools that take your entire project and package it into something the browser can efficiently load and execute.

### Why Bundling is Needed

```
1000 separate files in a folder
      ↓  (combine + compress, like a ZIP)
   1 optimized output file
      ↓
Given to the browser  →  JS engine executes it
```

Loading 1000 individual files over the network (each with its own HTTP request/parse overhead) is slow. Bundling merges and compresses them into far fewer files, which the browser can load and run efficiently.

### Bundlers by Phase

| Bundler | Used During | Why |
|---|---|---|
| **esbuild** | Development (`npm run dev`) | Extremely fast (written in Go), ideal for near-instant dev server startup and HMR |
| **Rollup** | Production build / after deployment | Prioritizes highly optimized, tree-shaken output over raw speed — better suited for a one-time production build |

## 2. `useState` Recap

`useState` gives a component the ability to:
1. **Maintain state** (a value that persists across renders).
2. **Re-render the component** whenever that state changes.
3. **Provide interactivity** — UI that responds to user actions.

```js
const [count, setCount] = useState(0);
// 0 here is a PRIMITIVE type

const [user, setUser] = useState({ name: "John", age: 30 });
// { ... } here is a REFERENCE type — user holds a reference (memory address) to the object, not the object's contents directly
```

This distinction — **primitive vs reference type** — is the core of this module, because it directly affects *when* React decides to re-render.

---

## 3. Primitive State — Straightforward Updates

```js
onClick = () => setCount(count + 1);
```

- `count + 1` always produces a **brand-new primitive value**.
- New value ≠ old value → React re-renders, and the updated `count` is visible immediately.

**Primitives are immutable:**
- A primitive's original value can never be changed or mutated in place — any "change" actually creates a completely new value.
- The variable stores the **actual value** directly, not a reference to it.
- Because of this, `useState` with primitives behaves predictably: any calculated new value is inherently a new, comparably-different value, so React reliably detects the change and re-renders.

---

## 4. Reference State — The Tricky Part

Now consider an object in state:

```js
const [user, setUser] = useState({ name: "John", age: 30 });
```

`user` doesn't hold the object's data directly — it holds a **reference** (think of it like an address, e.g. `xyz`) pointing to where that object lives in memory.

### 4.1 Mutating Directly (No `setUser` Call)

```js
onClick = () => { user.name = "Doe"; };
```

- **Does the value change?** Yes — in raw JavaScript terms, `user.name` genuinely becomes `"Doe"` in memory.
- **Does the component re-render?** **No.**
- **Why not?** Because `setUser` was never called. React has no built-in way to "see" that you mutated an object directly — it only knows to re-render when its setter function is explicitly invoked. The state technically changed, but React is completely unaware.

### 4.2 Calling `setUser(user)` After Mutating

```js
user.name = "Doe";
setUser(user);
```

- **Does the UI now show "Doe"?** **NO.**
- **Why not?** Because `user` still points to the **exact same reference** (`xyz`) as before — you mutated the object in place rather than creating a new one. React compares the new state to the old state by reference (`Object.is()`), and since it's literally the *same* object reference, React sees no difference and **bails out** of re-rendering (see the bail-out rule from earlier notes).

> **Key idea:** a reference type never "stores a new value" in place — the reference itself stays the same unless you explicitly create a new object.

### 4.3 The Fix — Passing a New Reference

```js
setUser({ name: "Doe" });   // brand-new object → new reference, e.g. pqrs
```

- This creates an entirely **new object in memory** with a **new reference**.
- React compares old reference (`xyz`) vs new reference (`pqrs`) → they differ → React re-renders, and the UI correctly shows `"Doe"`.

**This is why it "re-renders again and again" whenever you pass a freshly created object** — every call produces a new reference, so React never bails out.

### 4.4 Putting It Together

```js
onClick(userButton) → user.name = "Doe";   // mutated, but NOT rerendered (no setUser call)
onClick(countButton) → setCount(count + 1); // setCount IS called → whole App rerenders
```

Since React re-renders the **entire component function** (not just the piece that changed), the moment *any* state setter fires — even for `count` — the component function re-runs from top to bottom. At that point, `user.name` (already mutated to `"Doe"` in memory earlier) is now read fresh during this re-render and **will appear updated on screen**, even though no `setUser` was ever called.

> The mutation was real in memory the whole time — it just wasn't *painted* to the UI until some re-render (triggered by any state setter) happened to run and read the already-mutated value.

## 5. Multiple `useState` Calls & Where Values Live

```js
const [user, setUser] = useState(referenceType);
const [count, setCount] = useState(referenceType);
const [rename, setRename] = useState(referenceType);
const [product, setProduct] = useState(referenceType);
```

- Each `useState` call manages its **own independent piece of state**.
- Updated values are conceptually held in the component's **lexical scope** for that render — each render is a fresh function call with its own snapshot of state values, supplied by React.
- **Every one of these states requires its own re-render trigger** (i.e., calling its respective setter) to reflect changes on screen — updating one state doesn't "notify" another.

## 6. Primitive State — A Subtle Immutability Point

```js
const [count, setCount] = useState(0);  // primitive
const [user, setUser] = useState(0);    // primitive (just a number here, for comparison)

onClick(countBtn) → setCount(count + 1);  // re-renders
onClick(userBtn)  → user++;               // does NOT call setUser at all
```

- Clicking the count button works fine — `setCount` is explicitly called, so React re-renders.
- Clicking the user button does `user++` directly on the variable — but **no setter is called**, so React doesn't know anything changed, and there's **no re-render**.
- Even conceptually, "behind the scenes" `user` becomes `1` in that local variable — but this new value is **never shown**, because:
  - **Primitive values in `useState` are immutable.** You cannot reach into a primitive and update it in place the way you can mutate an object's property — any change to a primitive is really the creation of a brand-new value, and that new value only makes it into React's state (and the UI) if you pass it through the setter function.

### Immutable vs Mutable — Summary

| | Immutable (Primitives) | Mutable (Reference Types) |
|---|---|---|
| Examples | numbers, strings, booleans | objects, arrays |
| Can the original value be changed in place? | No — any "change" creates a new value | Yes — properties can be changed directly on the same reference |
| What's stored in the variable | The **actual value** itself | A **reference** (pointer) to the value in memory |
| Re-render trigger | New value automatically differs from old → safe with direct `setState(newValue)` | Must create a **new object/array reference** (e.g. spread `{ ...obj }`) — mutating in place won't trigger a re-render |

## 7. Code Walkthrough

```jsx
import React, { useState } from 'react'

const App = () => {
  const [count, setCount] = useState(0);
  console.log(count);

  const [user, setUser] = useState({ name: "Raman" });
  console.log(user);

  return (
    <div className='app'>
      <span>{count}</span>
      <h1>VariableCount</h1>
      <button className='inc' onClick={() => setCount(count + 1)}>Increment</button>
      <h2 className="userText">{user.name}</h2>
      <button className='userName' onClick={() => setUser({ ...user, name: "Baman" })}>†</button>
    </div>
  )
}

export default App
```

### Breakdown

- **`count` (primitive state)** — `setCount(count + 1)` always passes a new number → straightforward re-render every click, `<span>{count}</span>` updates correctly.
- **`user` (reference state)** — initial value `{ name: "Raman" }`.
- The `userName` button calls:
  ```js
  setUser({ ...user, name: "Baman" })
  ```
  - `{ ...user, name: "Baman" }` **spreads** all existing properties of `user` into a **brand-new object**, then overwrites `name` with `"Baman"`.
  - This produces a **new reference** every time — even though conceptually only `name` changed, the whole object is new in memory.
  - Because the reference differs from the previous `user` state, React detects the change and **re-renders**, so `<h2 className="userText">{user.name}</h2>` correctly updates to `"Baman"`.
  - Contrast this with `user.name = "Baman"; setUser(user);` — that would **not** update the UI, because `user` would still be the *same* reference (per Section 4.2 above).

### Styling Notes (`index.css`)

- `.userText` is absolutely positioned top-left, styled in orange (`#FF4500`), showing the current `user.name`.
- `.userName` is the circular orange **†** button, absolutely positioned top-right, used to trigger the object-reference update.
- `.inc` is the green Increment button; `span` (showing `count`) is large (50px) and green (`#50C878`) as the visual focal point.
- Both buttons share a small `scale(0.95)` press animation via `:active` for tactile feedback.

## Quick Summary

```
Primitive state (numbers, strings, booleans):
  - Immutable — any change = a new value automatically
  - setState(newValue) just works, since newValue is inherently different

Reference state (objects, arrays):
  - Mutable — properties CAN be changed in place, but React won't notice
  - Direct mutation (obj.prop = x) → NOT detected → no rerender
  - setState(sameReference) → NOT detected (same reference) → no rerender
  - setState({ ...obj, prop: x }) → NEW reference → detected → rerender ✅

Rule: React always compares old vs new state BY REFERENCE for objects/arrays.
      Always create a new object/array (spread syntax) when updating reference state.
```
