# React + Vite

## 1. Vite — Build Tool

Vite is a modern front-end build tool that gives you a fast development server and an optimized production build.

- **Dev server** — uses **esbuild** (written in Go) to pre-bundle dependencies and serves your source files over native **ES Modules**, so there's almost no bundling step during development. This is why `npm run dev` starts almost instantly, even on large projects.
- **Production build** — uses **Rollup** to bundle and tree-shake your code into optimized static assets.

| Piece | Tool | Role |
|---|---|---|
| Bundler | Rollup (build) / esbuild (dev) | Combines many files into fewer output files, tree-shakes unused code |
| Compiler / Transpiler | Babel (or esbuild/SWC) | Converts modern JS/JSX into browser-compatible JS |
| Next-gen bundler | **Rolldown** | A Rust-based bundler (built on the Rollup API) that Vite is migrating to, aiming to replace both Rollup and esbuild for a single, faster, unified bundling engine |

> **Bundler vs Compiler**: A bundler decides *how files are combined and what's kept*. A compiler/transpiler decides *how the syntax is rewritten* (e.g., JSX → `React.createElement` calls, or modern JS → ES5).

---

## 2. What Happens on `npm run dev`

When you run `npm run dev`, Vite's dev server pipeline roughly does:

1. **Compile** — JSX/TS syntax is transpiled to plain JS (via Babel/esbuild/SWC).
2. **Transform on demand** — Vite only transforms the modules the browser actually requests, not the whole app upfront.
3. **Bundle (dev-only, partial)** — third-party dependencies are pre-bundled with esbuild for performance; your own source files are served as native ES modules.
4. **Execute** — the browser loads and executes the modules via `<script type="module">`.
5. **Hot Module Replacement (HMR)** — when you save a file, Vite pushes just the changed module to the browser over a WebSocket connection, instead of reloading the whole page.
6. **Virtual DOM diffing** — React compares the new Virtual DOM tree with the previous one (reconciliation).
7. **Real DOM update** — only the actual changed nodes are updated in the browser's real DOM, which is what makes UI updates fast.

## 3. JSX

**JSX = JavaScript XML** — a syntax extension for JavaScript that lets you write HTML-like markup directly inside JS code.

- JSX is **not** valid browser JavaScript by itself — it must be compiled (by Babel or esbuild) into regular `React.createElement()` calls before the browser can run it.
- Example:
  ```jsx
  const element = <h1>Hello World</h1>;
  // compiles to:
  const element = React.createElement('h1', null, 'Hello World');
  ```
- JSX allows embedding JS expressions using curly braces: `<h1>{userName}</h1>`.
- Every JSX expression must return a **single root element** (or use a Fragment `<>...</>`).

## 4. Components

React apps are built from **components** — reusable, self-contained pieces of UI. The modern standard is **Functional Components**.

**Naming rules:**
- The **file name's first letter should be uppercase** (convention, e.g. `About.jsx`).
- The **function name's first letter must be uppercase** — this is how React tells apart a custom component (`<About />`) from a regular HTML tag (`<div>`). Lowercase names are treated as native DOM elements.

**Usage:**
- Components are called/used just like HTML elements:
  ```jsx
  <About />        // self-closing, no children
  <About>Text</About>  // with children
  ```

---

## 5. Functions vs. Functional Components

| | Regular Function | Functional Component |
|---|---|---|
| Definition | `(parameters) => { ... }` | `(props) => { ... }` |
| Invocation | `myFunction(arguments)` — a direct function call | `<MyComponent prop="value" />` — JSX syntax, compiled to `React.createElement(MyComponent, {...props})` |
| Return value | Any value (number, string, object, etc.) | JSX (i.e., a description of UI to render) |
| Parameter | Can accept anything | Conventionally accepts a single `props` object |

A functional component is really just a plain JavaScript function — React just expects it to return JSX and calls it with a `props` object as its argument.

## 6. Props

**Props (short for "properties")** are how data is passed from a parent component into a child component.

- Props are **read-only** (immutable from the child's perspective) — a component should never modify its own props.
- By default, if no props are passed, `props` is an **empty object** (`{}`), not `undefined`.
- Example:
  ```jsx
  function About(props) {
    return <h1>Hello, {props.name}</h1>;
  }

  <About name="John" />   // props = { name: "John" }
  <About />               // props = {}
  ```
- Props flow **one-way**: parent → child (this is called "unidirectional data flow").

---

## 7. `React.createElement()`

JSX is syntactic sugar over `React.createElement()`. Its signature is:

```js
React.createElement(type, props, ...children)
```

Example:
```js
React.createElement('h1', {}, 'Hello World')
```

| Argument | Meaning |
|---|---|
| `'h1'` | The element type (HTML tag name or a component reference) |
| `{}` | The **props object** — empty here since no attributes were passed |
| `'Hello World'` | The children (can be text, elements, or an array of elements) |

This call returns a plain JS object (a "React element") describing what should appear on screen — not an actual DOM node. React's reconciler later uses this object to figure out what real DOM changes are needed.

---

## Quick Summary Flow

```
JSX  --(Babel/esbuild)-->  React.createElement() calls
   --> React elements (JS objects)
   --> Virtual DOM tree
   --> Reconciliation (diffing old vs new tree)
   --> Real DOM updates
```
