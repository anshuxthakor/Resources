# React — State Batching & Functional Updates

## 1. Recap: What `setState` Does

The setter function returned by `useState` (e.g. `setCount`) has two jobs:

1. **Updates the state value.**
2. **Triggers a re-render** of the component that owns the state.
3. **Bail-out rule:** if the *same* value is passed again, React skips the re-render (see previous notes on `useState`).

This module covers what happens when you call the setter **multiple times in a single event handler**.

---

## 2. The Problem — Calling `setState` Repeatedly

```js
let count = 0;
// h1 -> {count}

button.onClick = () => {
  setCount(count + 1);
  setCount(count + 1);
  setCount(count + 1);
  // ... imagine this repeated 1000 times
};
```

**Naive expectation:** calling `setCount` 1000 times should re-render the component 1000 times, and the final count should be `1000`.

**Actual output:** `count` becomes `1`, and the component **re-renders only once**.

### Why?

Every call above is `setCount(count + 1)`, where `count` is the **same stale value** captured from the current render's closure (it's `0` in every single call within that one event handler run — `count` doesn't magically update *between* these synchronous lines). So effectively you're calling `setCount(0 + 1)` a thousand times in a row — it's the same instruction repeated, not an accumulating one.

---

## 3. Batching — The Real Reason for "Only 1 Re-render"

> **Batching** is React's optimization where multiple `setState` calls that happen within the same event handler (or synchronous block of code) are **grouped together** and processed as a **single update**, causing only **one re-render** at the end — instead of one re-render per call.

```
setCount(count + 1)   ┐
setCount(count + 1)   ├──►  React batches these  ──►  ONE re-render
setCount(count + 1)   ┘        (event loop tick)
```

**Why does React do this?** Re-rendering (and the underlying diffing + real DOM commit) is expensive. If React re-rendered on *every single* `setState` call, a handler that calls `setState` 1000 times would trigger 1000 wasted render cycles before the user even sees anything. Batching means React waits until the current execution context (the event handler) finishes, looks at *all* the state updates that were queued, and then performs just **one** render + commit with the final values.

### Automatic Batching (React 18+)

- Before React 18, batching only reliably happened inside **React event handlers** (like `onClick`). Updates inside `setTimeout`, promises, or native event listeners would each trigger their own re-render.
- Since **React 18**, batching is **automatic everywhere** — including timeouts, promises, and native event handlers — not just inside React's synthetic event handlers. This is one of React 18's key performance improvements.

---

## 4. The Fix — Functional Updates (`prevState => ...`)

Instead of reading the outer `count` variable (stale, closure-captured), you can pass a **function** to the setter. React guarantees this function always receives the **most recent, up-to-date state** — even across multiple queued updates within the same batch.

```js
setCount(prevCount => prevCount + 1);
setCount(prevCount => prevCount + 1);
setCount(prevCount => prevCount + 1);
```

**Output:** `count` becomes `3`, and the component still re-renders only **once**.

### Why This Works

- `setState(newValue)` → React just stores `newValue` directly; each call overwrites the last, ignoring the others.
- `setState(prevState => ...)` → React queues these as an **update function pipeline**. When it's time to actually compute the final state, React runs each function **in order**, passing the *result of the previous one* as the input to the next:

```
Initial state: 0
setCount(prev => prev + 1)  →  0 + 1 = 1
setCount(prev => prev + 1)  →  1 + 1 = 2
setCount(prev => prev + 1)  →  2 + 1 = 3
Final state after batch: 3
Re-renders: 1 (still batched — just the computed value is now correct)
```

This is why the functional updater form is often called the **"previous state" pattern** — inside `setFunction`, you get access to the actual latest/current state at the moment React processes the queue, not the stale value from your closure.

> **Rule of thumb:** whenever your new state depends on the *previous* state, always use the functional updater form `setState(prev => ...)` instead of `setState(someExpressionUsingOuterVariable)`.

---

## 5. Direct Value vs Functional Updater — Comparison

| | `setCount(count + 1)` × 3 | `setCount(prev => prev + 1)` × 3 |
|---|---|---|
| What's passed | A pre-computed value based on stale closure `count` | A function that computes from the *actual* latest queued state |
| Result after 3 calls | `1` (each call overwrites with the same stale computation) | `3` (each call builds on the previous queued result) |
| Re-renders | 1 (batched) | 1 (batched) |
| Use when | New state does **not** depend on the previous state (e.g. `setName("John")`) | New state **depends** on the previous state (increments, toggles, appending to lists, etc.) |

## 6. Code Walkthrough

```jsx
import React, { useState } from "react";

const App = () => {
  console.log("App component rendered");
  const [count, setCount] = useState(0);
  console.log(count);

  return (
    <div className="app">
      <span>{count}</span>
      <h1>VariableCount</h1>
      <button
        className="inc"
        onClick={() => {
          // Batching state updates in React 18
          setCount((prevCount) => prevCount + 1);
          setCount((prevCount) => prevCount + 1);
          setCount((prevCount) => prevCount + 1);
        }}
      >
        Increment
      </button>
    </div>
  );
};

export default App;
```

**What happens on one click of "Increment":**

1. The `onClick` handler runs synchronously, queuing **three** functional updates: `prev => prev + 1` (×3).
2. React does **not** re-render after each call — it batches all three within this single event handler execution.
3. Once the handler finishes, React processes the queue in order: `0 → 1 → 2 → 3`.
4. React re-renders **once** with the final value, `3`.
5. Console output confirms this: `"App component rendered"` and `console.log(count)` each print **once** per click (not three times) — proof that batching collapsed three `setCount` calls into a single render cycle.
6. On the *next* click, the same thing happens again: `3 → 4 → 5 → 6`, one re-render, console logs fire once.

---

## Quick Summary

```
Multiple setState calls in one event handler
   → React queues them instead of rendering immediately
   → Batching: all updates processed together
   → ONE render phase + ONE commit phase at the end

setState(value)              → uses stale/closure value → overwrites, doesn't accumulate
setState(prev => prev + 1)   → uses real queued latest state → accumulates correctly

React 18+: batching is automatic everywhere (event handlers, promises, timeouts, native listeners)
```
